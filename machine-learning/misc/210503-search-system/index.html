1. Serving System

  • User : 사용자가 웹 브라우저를 통해 https로 요청을 넣으면 NXS가 html로 응답

  • NXS : 통검서버, 사용자의 요청에 html로 웹 결과를 제공 (relay, (이를 브라우저가 해석해서 예쁘게 보여줌)

    1. 검색결과제공 (user와는 https, 뒷단서버와는 http)

    2. Collection Ranking (통검에서는 컬렉션간 랭킹만을 고려, ‘명탐정코난’ 에 대하여는 video가 1등으로)

      • FAS : 사용자 반응 feedback : ‘명탐정코난’ 키워드는 video가 많이 눌려지더라

      • Rule Based : FAS와 무관하게 강제적으로 검색결과 제공 (이나영 => 인물정보를 맨 위로 ), 사실 rule based가 큰 비중

    3. 로그 저장

      • 노출로그

      • 클릭로그

  • PRS : 검색 결과 화면을 구성해주는 역할을 담당해주는 서버 , XML 혹은 JSON 으로 형태로 제공되는 검색결과를 HTML로 바꿔서 출력

  • SAS (Search Application Server) : 한단위의 검색 요청을 온전히 수행하여 검색된 결과를 XML, JSON 으로 제공

    • 모든 서비스 단위의 기본 검색 API를 제공 (블로그 서비스를 만드는데 검색 API가 필요하다?!)

    • 사내에서 주로 사용되는 검색엔진은 NEXUS (C++) & NPP (C-lang)

    • 웹서버를 C로 구성하는것은 네이버에서는 일반적이나 외부에서는 그렇지 않음

    • 모델링 : SLIM : 검색 랭킹의 핵심은 Relevance (Score) = Similarity + Quality

2. Modeling & Search Engineering

  • Collection : Documents set으로 이루어짐

  • Index : 색인, 주어진 doc에서 핵심단어 (형태소분석) 를 추출하여 색인, (TF-IDF score가 저장)

  • Ranking Model : 문서 검색 결과를 rlevance 가 높은 순서대로 제공하는것, 즉 모든 문서의 rel값을 계산한 뒤 각장 큰 값으로 제공

    • Relevance = similarity + quality (BM25 Algorithm)

    • Relevance 는 색인할때 추가해야하기에 이 값을 바꾸려면 색인을 다시해야함
