9.1 Designing an algebra first (215)

  • Recall that we defined an algebra as a collection of functions operating over some data type(s), along with a set of laws specifying relationships between these function

    • In past chapters, we moved rather fluidly between

      • inventing functions in our algebra,

      • refining the set of functions,

      • and tweaking our data type representations.

  • In this chapter, We’ll start with the algebra (including its laws) and decide on a representation later. Let’s call this approach, algebraic design.

    • Let’s begin!

      • Library for what? : Parsing library that operate on strings as input.

      • what to parse? : start with parsing various combinations of repeated letters and gibberish words like “abracadabra” and “abba”.

    • (1) Let’s start with the simplest of parsers: one that recognizes the single character input ‘a’ : invent a combinator for the task—char:

      • def char(c: Char): Parser[Char]

    • (2) running a parser shouldn’t simply yield a yes/no response; if it succeeds, we want to get a result that has some useful type, and if it fails, we expect information about the failure.

      • extension [A](p: Parser[A]) def run(input: String): Either[ParseError, A]

    • (3) specifying an interface that happens to make use of two types => Let’s make this explicit with a trait

      • trait Parsers[ParseError, Parser[+_]]:
        • Parser[+_] : Scala’s syntax for a type parameter that is itself a type constructor.

    • (4) What if we want to recognize the string ? (not only char)

      • def string(s: String): Parser[String]

    • (5) what if we want to recognize either the string “abra” or the string “cadabra”? We could add a very specialized combinator for it:

      • def orString(s1: String, s2: String): Parser[String]

    • (6) But choosing between two parsers seems like something that would be more generally useful, regardless of their result type, so let’s make this polymorphic: orString => or

      • extension [A](p: Parser[A]) def or(p2: Parser[A]): Parser[A]

      • we can give this or combination nice infix syntax, like s1|s2

// (3) specifying an interface that happens to make use of two types (Parser, ParseError)
trait Parsers[ParseError, Parser[+_]]:
  // (1) base combinator: char('a') parser will succeed only if the input is exactly the character 'a', and it will return that same character 'a' as its result.
  def char(c: Char): Parser[Char]  
  // (4) What if we want to recognize the string?
  def string(s: String): Parser[String]
  // (2) get a result or information about the failure
  extension [A](p: Parser[A]) 
    def run(input: String): Either[ParseError, A] // HW ) When we run the parser => it will return the input or ParseError
    // (5) polymorphic
    infix def or(p2: Parser[A]): Parser[A]
    // (6)  The infix modifier allows or to be used as an operator.
    def |(p2: Parser[A]): Parser[A] = p.or(p2)

  • We can now recognize various strings, but we don’t have a way of talking about repetition.

    • how would we recognize three repetitions of our string("abra") | string("cadabra") parser? Once again, let’s add a combinator for it
      • extension [A](p: Parser[A]) def listOfN(n: Int): Parser[List[A]]

  • At this point, we’ve just been collecting required combinators,

    • but we haven’t tried to refine our algebra into a minimal set of primitives,
    • and we haven’t talked much about more general laws.

  • We’ll start doing this next


9.2 A possible algebra

  • (1) First, let’s consider the parser that recognizes zero or more repetitions of the character ‘a’ and returns the number of characters it has seen. - many

    • extension [A](p: Parser[A]) def many: Parser[List[A]]

  • (2) we need a Parser[Int] that counts the number of elements.

    • We could change the many combinator to return a Parser[Int],

    • But It is better to introduce another combinator that should be familiar by now—map

      • extension [A](p: Parser[A]) def map[B](f: A => B): Parser[B]

  • (3) We can now define our parser like this:

    • val numA: Parser[Int] = char('a').many.map(_.size)

    • ex) numA.run("aaa")gives Right(3) and numA.run("b") gives Right(0).

  • (4) we expect map to be structure preserving,

    • merely transform the result value if the Parser was successful.

    • No additional input characters should be examined by map,

    • a failing parser can’t become a successful one via map,

  • (5) Let’s formalize this by stipulating the now-familiar law:

    • p.map(a => a) == p

  • (6) We could put this law in a documentation comment, but in the preceding chapter, we developed a way of making our laws executable.

trait Parsers[ParseError, Parser[+_]]:
  ...
  object Laws:
    def equal[A](p1: Parser[A], p2: Parser[A])(in: Gen[String]): Prop =
      Prop.forAll(in)(s => p1.run(s) == p2.run(s))
    def mapLaw[A](p: Parser[A])(in: Gen[String]): Prop =
      equal(p, p.map(a => a))(in)

  • This will come in handy later when we test that our implementation of Parsers behaves as we expect.

  • When we discover more laws later on, you’re encouraged to write them out as actual properties inside the Laws object.

  • And now we can refine combinators we made from 9.1

    • Incidentally, now that we have map, we can implement char in terms of string:

      • def char(c: Char): Parser[Char] = string(c.toString).map(_.charAt(0))

    • Similarly, another combinator, succeed, can be defined in terms of string and map:

      • def succeed[A](a: A): Parser[A] = string("").map(_ => a)

    • This parser always succeeds with the value a, regardless of the input string (since string("") will always succeed, even if the input is empty). Does this combinator seem familiar to you? We can specify its behavior with a law:

      • succeed(a).run(s) == Right(a)

9.2.1 Slicing and nonempty repetition

  • Slicing

    • The combination of many and map certainly lets us express the parsing task of counting the number of ‘a’ characters,

      • but it seems inefficient to construct a List[Char] only to discard its values and extract its length.

        • slice : It would be nice if we could run a Parser purely to see what portion of the input string it examines

          • extension [A](p: Parser[A]) def slice: Parser[String]

    • With slice, our parser that counts ‘a’ characters can now be written as :

      • char('a').many.slice.map(_.size).

    • slice is primitive : since it have to have access to the internal representation of the parser.

      • parser p.many.map(_.size) will generate an intermediate list when run, p.many.slice.map(_.size) will not.

  • non-empty repetition

    • What if we want to recognize one or more ‘a’ charactors - many1

      • p.many1 is just p followed by p.many

    • we need some way of running one parser followed by another, assuming the first is successful, and returning the product of their results.

extension [A](p: Parser[A])
  def product[B](p2: Parser[B]): Parser[(A, B)]
  def **[B](p2: Parser[B]): Parser[(A, B)] = product(p2) // Operator alias for product, where a ** b delegates to a.product(b)

9.3 Handling context sensitivity

  • Let’s take a step back and look at the primitives we have so far:

    • string(s)—Recognizes and returns a single String

    • p.slice—Returns the portion of input inspected by p if successful

    • succeed(a)—Always succeeds with the value a

    • p.map(f)—Applies the function f to the result of p, if successful

    • p1.product(p2)—Sequences two parsers, running p1 and then p2, and returns the pair of their results if both succeed

    • p1 or p2—Chooses between two parsers, first attempting p1 and then p2 if p1 fails

  • Using these primitives, we can express repetition (many, listOfN), as well as combinators like char

    • but what can’t we express yet?

  • Context-sensitive grammar

    • Suppose we want to parse a single digit followed by many ‘a’ characters, like "0", "1a", "2aa", "4aaaa"

    • We want to run the first parser and then do a listOfN using the number extracted from the first parser’s result.

    • It can’t be expressed with product (the second parser depends on its context)

    • In past chapters, we dealt with these limitations by introducing a new primitive - flatmap:

      • flatMap enables context-sensitive parsing and allows us to implement map

      • extension [A](p: Parser[A]) def flatMap[B](f: A => Parser[B]): Parser[B]

  • With flatMap, instead of the less-general map and product,

    • we can parse not just arbitrary context-free grammars, like JSON,

    • but context-sensitive grammars as well, including extremely complicated ones, like C++ and PERL.


9.4 Writing a JSON parser

  • Let’s write that JSON parser now
    • Our JSON parser doesn’t need to know the internal details of how parsers are represented
    • We can simply write a function that produces a JSON parser using only the set of primitives we’ve defined and any derived combinators.
    • This might seem like a peculiar thing to do, However, we’ll proceed because in FP it’s common to define an algebra and explore its expressiveness without having a concrete implementation.
    • Especially during the design phase of a library, it can be much easier to refine an algebra without having any particular implementation, and part of our goal here is getting you comfortable with this style of working.

9.4.2 A JSON parser (Exercies 9.9)

  • You’ll be creating a Parser[JSON] from scratch using the primitives we’ve defined. You don’t need to worry (yet) about the representation of Parser.

  • Use the skills you’ve been developing throughout this book, and have fun!

package fpinscala.answers.parsing

// We need a representation for a parsed JSON document.
enum JSON:
  case JNull
  case JNumber(get: Double)
  case JString(get: String)
  case JBool(get: Boolean)
  case JArray(get: IndexedSeq[JSON])
  case JObject(get: Map[String, JSON])

object JSON:
  def jsonParser[Parser[+_]](P: Parsers[Parser]): Parser[JSON]=
    import P.*

    def token(s: String) = string(s).token

    def array: Parser[JSON] = (
      token("[") *> value.sep(token(",")).map(vs => JArray(vs.toIndexedSeq)) <* token("]")
    ).scope("array")

    def obj: Parser[JSON] = (
      token("{") *> keyval.sep(token(",")).map(kvs => JObject(kvs.toMap)) <* token("}")
    ).scope("object")

    def keyval: Parser[(String, JSON)] = escapedQuoted ** (token(":") *> value)

    def lit: Parser[JSON] = (
      token("null").as(JNull) |
      double.map(JNumber(_)) |
      escapedQuoted.map(JString(_)) |
      token("true").as(JBool(true)) |
      token("false").as(JBool(false))
    ).scope("literal")

    def value: Parser[JSON] = lit | obj | array

    (whitespace *> (obj | array)).root

9.5 Error reporting

  • So far, we haven’t discussed error reporting at all.

    • We’ve focused exclusively on discovering a set of primitives that let us express parsers for different grammars,

    • but besides just parsing a grammar, we want to be able to determine how the parser should respond when given unexpected input.

  • Even without knowing what an implementation of Parsers will look like, we can reason abstractly about what information is being specified by a set of combinators.

9.5.1 A possible design

  • We can introduce a primitive combinator for this—label:

    • extension [A](p: Parser[A]) def label(msg: String): Parser[A]

    • The intended meaning of label is that if p fails, its ParseError will somehow incorporate msg.

  • but we’d like our parse error to also tell us where the problem occurred. Let’s tentatively add this to our algebra: Location

    • concrete representation for Location that includes the full input, an offset into this input, and the line and column numbers, which can be computed lazily.
case class Location(input: String, offset: Int = 0):
  lazy val line = input.slice(0, offset + 1).count(_ == '\n') + 1
  lazy val col = input.slice(0, offset + 1).lastIndexOf('\n') match
       case -1 => offset + 1
       case lineStart => offset - lineStart

def errorLocation(e: ParseError): Location // Extracts a concrete Location from an abstract ParseError
def errorMessage(e: ParseError): String // Extracts a concrete message from an abstract ParseError

9.5.2 Error nesting

  • This notion is still a bit fuzzy; if we have a | b, and both parsers fail on the input, which location and label(s) are reported?

  • it seems wrong to assume that one level of error reporting will always be sufficient. Let’s therefore provide a way to nest labels: - scope

    • extension [A](p: Parser[A]) def scope(msg: String): Parser[A]

    • Unlike label, scope doesn’t throw away the label(s) attached to p—it merely adds additional information in the event that p fails.

  • Let’s modify the functions that pull information out of a ParseError. Rather than containing just a single Location and String message, we should get a List[(Location, String)]:

    • case class ParseError(stack: List[(Location, String)])
    • ParseError is a stack of error messages indicating what the Parser was doing when it failed.

  • Now we’re giving the Parsers implementation all the information it needs to construct nice, hierarchical errors.


9.7 Conclusion

  • In this chapter, we introduced algebraic design, an approach to writing combinator libraries, and we used it to design a parser library and implement a JSON parser.

    • Along the way, we discovered a number of combinators similar to what we saw in previous chapters, and these were again related by familiar laws.

    • In part3, we’ll finally understand the nature of the connection between these libraries and learn how to abstract over their common structure.

  • Summary

    • A parser converts a sequence of unstructured data into a structured representation.

    • A parser combinator library allows the construction of a parser by combining simple primitive parsers and generic combinators.

    • Algebraic design is the process in which an interface and associated laws are designed first and then used to guide the choice of data type representations.

    • Judicious use of infix operators, either defined with symbols as in | or with the infix keyword as in or, can make combinator libraries easier to use.

    • The many combinator creates a parser that parses zero or more repetitions of the input parser and returns a list of parsed values.

    • The product combinator (or ** operator) creates a parser from two input parsers, which runs the first parser and, if successful, runs the second parser on the remaining input. The resulting value of each input parser is returned in a tuple.

    • The map and flatMap operations are useful for building composite parsers. In particular, flatMap allows the creation of context-sensitive parsers.

    • The label and scope combinators allow better error messages to be generated with parsing fails.

    • APIs can be designed by choosing primitive operations, building combinators, and deciding how those operations and combinators should interact.

    • API design is an iterative process, where interactions amongst operations, sample usages, and implementation difficulties all contribute to the process.
